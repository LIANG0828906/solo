## 3. 模块职责与数据流

### 3.1 模块划分原则
- 单一职责：每个模块只负责一件事
- UI 与逻辑分离：配色算法完全独立于 React
- 单向数据流：数据通过 Props 向下传递，事件通过回调向上传递

### 3.2 模块详解

#### App.tsx — 主应用组件
**职责**：
- 管理全局状态（当前颜色、已用颜色集合、图片信息、历史记录步数等）
- 协调各模块间的数据流转
- 组合 UI 布局

**核心状态**：
```
selectedColor: string       // 当前选中的颜色
usedColors: string[]        // 已使用的颜色集合
imageInfo: { width, height } | null  // 图片尺寸信息
canvasKey: number           // 画布刷新用 key
```

#### CanvasArea.tsx — 画布模块
**职责**：
- 图片加载与显示
- 像素级 flood fill 区域填充算法
- 波纹扩散动画效果
- 撤销/重做历史栈管理（最多 20 步）
- 画布网格背景渲染

**核心算法**：
- 基于扫描线的 flood fill 算法（性能优化）
- 边缘抗锯齿处理（1px 柔和过渡）
- 环形扩散动画（基于 requestAnimationFrame）

**历史栈设计**：
```
historyStack: ImageData[]   // 历史记录栈
currentIndex: number        // 当前状态索引
MAX_HISTORY: 20             // 最大历史记录数
```

#### ColorPalette.tsx — 调色板组件
**职责**：
- 展示 12 种预设颜色
- 自定义颜色选择器
- 当前选中颜色高亮

**Props**：
```
selectedColor: string
onColorSelect: (color: string) => void
```

#### ColorSchemeEngine.ts — 配色推荐引擎
**职责**：
- 纯函数模块，不依赖 React
- 输入：已用颜色数组
- 输出：6 套配色方案（3 套互补色 + 3 套类似色）

**核心算法**：
- HSL 色彩空间转换
- 互补色计算（色相 +180°）
- 类似色计算（色相 ±30°）
- 颜色映射与替换规则生成

**导出函数**：
```
generateColorSchemes(usedColors: string[]): ColorScheme[]
```

#### exportUtils.ts — 导出工具
**职责**：
- Canvas 转 PNG 数据
- 生成带时间戳的文件名
- 触发浏览器下载
- 加载动画回调

**导出函数**：
```
exportCanvasToPNG(canvas: HTMLCanvasElement, options?: ExportOptions): Promise<void>
```

## 4. 关键技术方案

### 4.1 像素级区域填充算法

采用 **扫描线 Flood Fill** 算法，相比递归式 flood fill 有更好的性能和栈安全性：

1. 从点击点获取目标色（透明/白色区域）
2. 沿水平方向扫描连续可填充像素
3. 记录上下相邻行的种子点
4. 迭代处理所有种子点
5. 边缘像素做 alpha 渐变实现抗锯齿

**性能优化**：
- 使用 `Uint32Array` 直接操作像素数据
- 避免重复访问已处理像素
- 边界检查提前退出

### 4.2 波纹扩散动画

实现方式：
- 使用离屏 Canvas 预计算填充区域
- 通过 `requestAnimationFrame` 逐帧绘制
- 每帧扩大填充半径，模拟波纹扩散
- 总时长 300ms，使用 ease-out 缓动

### 4.3 配色方案生成算法

**色彩空间**：使用 HSL 而非 RGB，便于色相/饱和度/明度的独立调节。

**互补色方案**：
- 对每个已用颜色，色相 +180° 得到互补色
- 调整饱和度和明度保持和谐
- 生成 3 种变体（不同饱和度/明度组合）

**类似色方案**：
- 对每个已用颜色，色相 ±30° 得到类似色
- 生成 3 种变体（不同色相偏移量）

**颜色映射**：
- 建立旧色 → 新色的一一映射
- 保留原始填充的位置和形状

### 4.4 撤销重做机制

采用 **快照式历史记录**：
- 每次填充操作前保存当前 ImageData
- 撤销：回退到上一个快照，播放反向收缩动画
- 重做：前进到下一个快照，播放扩散动画
- 超过 20 步时丢弃最早的记录

## 5. 性能保障

### 5.1 填充性能
- 目标：点击到动画完成 ≤ 400ms
- 策略：
  - 扫描线算法减少像素遍历次数
  - 动画使用 CSS transform + opacity 而非逐像素重绘
  - 大图片时适当降低动画精度

### 5.2 配色计算性能
- 目标：配色推荐计算 ≤ 200ms
- 策略：
  - 纯函数计算，无 DOM 操作
  - 颜色数量有限（通常 < 12 种）
  - 计算量与图片尺寸无关

## 6. 动画系统

### 6.1 动画列表

| 动画名称 | 时长 | 触发时机 | 实现方式 |
|----------|------|----------|----------|
| 图片淡入缩放 | 400ms | 图片加载完成 | CSS transition |
| 波纹扩散填充 | 300ms | 点击填充 | Canvas + rAF |
| 反向波纹收缩 | 300ms | 撤销操作 | Canvas + rAF |
| 整体颜色切换 | 500ms | 应用配色方案 | Canvas 双层叠加 + opacity |
| 按钮悬停下划线 | 200ms | 鼠标悬停 | CSS transition |
| 按钮点击缩放 | 150ms | 按钮点击 | CSS transform |
| 导出加载旋转 | 300ms | 点击导出 | CSS animation |

### 6.2 动画原则
- 缓动函数：优先使用 ease-out 和 ease-in-out
- 性能：尽量使用 transform 和 opacity，避免触发重排
- 反馈：所有交互都要有即时的视觉反馈

## 7. 类型定义

```typescript
// 配色方案
interface ColorScheme {
  id: string;
  name: string;
  type: 'complementary' | 'analogous';
  colors: string[];
  colorMap: Map<string, string>; // 旧色 -> 新色
}

// 图片信息
interface ImageInfo {
  width: number;
  height: number;
  src: string;
}

// 画布操作类型
type CanvasAction = 'fill' | 'undo' | 'redo' | 'applyScheme';

// 导出选项
interface ExportOptions {
  filename?: string;
  onStart?: () => void;
  onComplete?: () => void;
}
```
